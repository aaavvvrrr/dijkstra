# 🚢 Maritime Route Finder (Spherical A* Raster Routing)

Cервис маршрутизации морских судов. Строит оптимальные, гладкие маршруты между любыми точками Мирового океана, обходя сушу и учитывая ограничения скоростного режима (основанного на данных AIS). 


## 💡 Архитектурное решение


** Алгоритм A* (Dijkstra) на регулярной 2D-сетке с учетом сферичности:**
1. **Memory Efficient (Float32):** Вместо просчета "всего мира от точки А" (FMM), A* направленно ищет путь к точке Б (эвристика Haversine). Это снижает потребление RAM с сотен гигабайт до десятков мегабайт.
2. **Local-Flat Projection:** Матрица сетки регулярна в градусах (Lat/Lon), но не в метрах. Алгоритм корректирует метрическую длину ячейки по оси X в зависимости от текущей широты (домножение на `cos(latitude)`). Это математически эквивалентно локальному шагу FMM на сфере.
3. **Гладкость (Chaikin's Algorithm):** Проблема "зигзагов" (D8-маршруты) решается не сложным градиентным спуском, который падает на границах суши, а геометрическим пост-процессингом (Алгоритм Чайкина), срезающим углы для получения усредненной плавной траектории.

## ⚙️ Как работает система (Шаг за шагом)

Система маршрутизации состоит из нескольких независимых этапов: от «очистки» сырых данных АИС до выдачи готового маршрута с оценкой рисков опоздания судна.

### Шаг 1. Препроцессинг (Подготовка данных)
Сырые данные АИС (координаты кораблей) очень «грязные»: в них есть разрывы, случайные выбросы скорости и погрешности GPS (когда корабли визуально «залезают» на берег).
Специальный Python-скрипт (`pipeline.py`) один раз подготавливает эти данные для бэкенда:
1. **Маска суши:** Алгоритм накладывает векторную карту океанов на треки АИС. Если по карте это суша, но там есть стоянки кораблей (порты) — алгоритм верит реальным трекам и «прорубает» воду к причалу.
2. **Сглаживание (Гауссиан):** Мелкие разрывы в треках заполняются усредненными значениями, чтобы маршрут не прерывался в открытом море.
3. **«Умная скорость» (Сигмоида):** Алгоритм штрафует зоны, где проплыл всего 1-2 корабля на огромной скорости. Он заставляет будущие маршруты «прилипать» к популярным и безопасным фарватерам (где плотность АИС высокая). 
*Итог шага: два чистых файла — растр «эффективной скорости» и растр «стандартного отклонения скорости (SD)»*.

### Шаг 2. Загрузка в память и Картографический сервер (Tile Server)
При запуске бэкенда (`main.py`) огромные матрицы данных загружаются в оперативную память **ровно один раз** (потребляя минимум RAM благодаря оптимизациям NumPy).
Чтобы пользователь в браузере мог видеть, где проходят скоростные трассы, а где мелководье, встроенный **Tile Server (rio-tiler)** «на лету» нарезает тяжелый 10-гигабайтный растр на крошечные прозрачные PNG-квадратики. Браузер скачивает только те квадратики, на которые сейчас смотрит пользователь. 

### Шаг 3. Умный старт (Авто-снэппинг)
Когда пользователь кликает на карту, он может случайно промахнуться и попасть в сушу (материк). 
Вместо того чтобы выдавать ошибку, ядро сервиса запускает алгоритм поиска в ширину (BFS — «раскручивающаяся спираль»). Он ищет ближайший к месту клика пиксель с реальной водой (где скорость > 0). Найдя его, алгоритм автоматически передвигает маркер старта или финиша в правильное место.

### Шаг 4. Поиск оптимального пути (Сферический A*)
Начинается магия маршрутизации:
1. Используется **Жадный алгоритм A\* (Weighted A*)**. В отличие от классических научных методов (FMM), которые просчитывают время до *всех* точек на Земле, наш алгоритм целенаправленно бежит только в сторону Финиша. Это ускоряет поиск в сотни раз.
2. **Учет кривизны Земли:** Сетка координат (широта/долгота) искажается у полюсов. Алгоритм корректирует длину каждого шага по оси X, умножая её на косинус текущей широты.
3. **Бесшовный мир:** Если плыть из Японии в США, алгоритм не упрется в край карты, а бесшовно перейдет через 180-й меридиан (Тихий океан) на другую сторону матрицы.
4. **Предохранитель (Таймаут):** Если пользователь попытался проложить маршрут в закрытое озеро (например, Каспийское море), алгоритм оборвет бессмысленный поиск через 15 секунд и вежливо сообщит, что путь прегражден сушей.

### Шаг 5. Сглаживание траектории
Матричный поиск выдает путь, идущий строгими «зигзагами» (в 8 направлениях). Корабли так не плавают.
Чтобы путь выглядел естественно, к найденным точкам применяется **геометрический алгоритм Чайкина** (срезание углов). Ломаная линия превращается в идеальную, гладкую, усредненную кривую морского пути.

### Шаг 6. Оценка рисков (Векторизованный Монте-Карло)
Бизнесу мало знать просто длину пути, ему нужно понимать риски задержки груза.
Для готового маршрута бэкенд извлекает среднюю скорость и её дисперсию (SD) на каждом участке пути. Затем с помощью математического аппарата NumPy генерируется **10 000 виртуальных заплывов** (симуляция Монте-Карло с использованием Гамма-распределения).
Это занимает доли секунды, но на выходе мы получаем:
* Оптимистичное время (5% заплывов были быстрее этого времени).
* Ожидаемое среднее время.
* Пессимистичное время (с вероятностью 95% корабль прибудет не позднее этого срока).

### Шаг 7. Выдача результата (API -> Frontend)
Сервер упаковывает всю информацию (включая посегментную разбивку дистанций и скоростей) в стандартизированный формат **GeoJSON**.
Браузер (Leaflet) получает этот файл, отрисовывает красивую пунктирную линию маршрута, фокусирует камеру на ней и выводит в боковую панель всю аналитику для конечного пользователя.

## 🛠 Технологический стек

- **Backend Logic:** Python 3.12+, NumPy (векторизация и матрицы), Shapely & Rasterio (растеризация векторной маски суши/океана), `heapq` (очередь с приоритетом).
- **API Layer:** FastAPI, Pydantic, Uvicorn.
- **Frontend / UI:** Vanilla JavaScript (ES6+), HTML5/CSS3 (CSS Variables, Flexbox), Leaflet.js (интерактивные карты).

## 🚀 Установка и запуск

### 1. Подготовка окружения
Убедитесь, что у вас установлен Python версии 3.12 или выше.

```bash
# Клонирование репозитория (или переход в папку проекта)
cd /path/to/project

# Создание и активация виртуального окружения
python -m venv .venv
source .venv/bin/activate  # Для Linux/MacOS
# .venv\Scripts\activate   # Для Windows

# Установка зависимостей
pip install -r requirements.txt
```

### 2. Данные
Убедитесь, что векторный файл маски океанов (`oceans-seas.geo.json`) находится в папке `app/static/`.

### 3. Запуск сервера
```bash
cd app
python main.py
```
Сервер запустится на порту `8000`. 
- **Пользовательский интерфейс (UI):** [http://localhost:8000](http://localhost:8000)
- **API Документация (Swagger):** [http://localhost:8000/docs](http://localhost:8000/docs)

При первом запуске сервер автоматически загрузит GeoJSON-маску и растеризует её в NumPy `float32` матрицу размером 900x1800 (шаг сетки 0.2 градуса).

## 🌐 Использование UI
1. Откройте интерфейс в браузере. Карта отрендерит полигоны морей (светло-синим цветом).
2. Кликните левой кнопкой мыши по воде для установки **Точки А (Старт)**.
3. Кликните второй раз по воде для установки **Точки Б (Финиш)**.
4. Нажмите кнопку **"Построить маршрут"**.
5. Результат (плавная пунктирная линия) появится на карте с указанием дистанции (км), времени (ч) и средней скорости.

## 🔌 API Спецификация

### Расчет маршрута
`POST /api/route`

**Request Body (JSON):**
```json
{
  "start_lon": -5.35,
  "start_lat": 36.14,
  "end_lon": 14.51,
  "end_lat": 35.89
}
```

**Response (GeoJSON - 200 OK):**
```json
{
  "type": "Feature",
  "properties": {
    "distance_km": 1745.32,
    "time_hours": 58.17,
    "avg_speed_kmh": 30.00
  },
  "geometry": {
    "type": "LineString",
    "coordinates": [
      [-5.35, 36.14],
      [-4.80, 36.20],
      /* ... остальные точки маршрута ... */
      [14.51, 35.89]
    ]
  }
}
```

**Возможные ошибки:**
- `400 Bad Request`: Старт или финиш находится на суше или вне зоны покрытия растра.
- `404 Not Found`: Маршрут не найден (точки изолированы друг от друга сушей).
- `500 Internal Server Error`: Внутренняя ошибка сервера.

## 🔮 План дальнейшего развития (Next Steps)
1. **Интеграция весов АИС:** Заменить текущую константную скорость (30 км/ч) на реальный композитный растр `float32`, объединяющий среднюю скорость судов (по типам, например, танкеры) и плотность треков (уменьшающий множитель для одиночных "выбросов" АИС).
2. **Повышение разрешения:** Увеличить размер сетки (например до 3600x7200, шаг 0.05 градуса) для более точного прохождения узких проливов (Босфор, Суэцкий канал). Производительность и потребление RAM бэкенда это легко позволяют.
3. **Автоматическое "прожигание" каналов:** Добавление узких векторных линий (проливов) в процесс `rasterio.features.rasterize`, чтобы они гарантированно имели ширину хотя бы в 1 пиксель.
